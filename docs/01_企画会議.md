## 1. コンセプト提案

**綾瀬**: 結論から言う。「ドラクエ・ポケモン世代の心臓を掴む、ピクセルアートRPG風のプログラミング学習アプリ」を作る。プレイヤーは勇者となってフィールドを歩き、「バグスライム」「ヌルポインタゴースト」「無限ループドラゴン」みたいなプログラミング概念を擬人化した敵とエンカウントする。戦闘はコマンドバトル風だが、攻撃の代わりに**プログラミングクイズに正解すると敵にダメージが入る**。正解を重ねて敵を倒すと経験値が入り、レベルアップする。これが骨格だ。

**白鳥**: 教育設計の観点から補強するわ。単なる「クイズにゲームの皮を被せたもの」は失敗する。重要なのは3点。第一に**即時フィードバック**——正解・不正解の理由をその場で1〜2行で示すこと。第二に**適切な難易度勾配**——フロー理論に基づき、敵が強くなるにつれて問題も段階的に難しくする。第三に**失敗の安全性**——不正解でもゲームオーバーで全ロストにはせず、「学び直せば再挑戦できる」構造にする。不正解時に解説が出ることで、間違いが罰ではなく学習機会になる。

**九条**: 技術側の制約を先に言っておく。初回リリースはサーバーレスの純粋なフロントエンドアプリにすべきだ。認証もDBも要らない。進捗はlocalStorageに保存。Canvas APIでピクセルアートを描画すれば、画像アセットすら不要でコードだけで完結できる。デプロイはVercelに静的サイトとして置くだけ。運用コストゼロ、表示は爆速。凝った技術より「1週間で出せて壊れない」ことが正義だ。

**桜庭**: 継続率の観点で一つ。ドラクエ的な「レベルアップのファンファーレ」体験は絶対に外せない。数字が上がる、HPバーが伸びる、新しいエリアが開く——この**可視化された成長**が学習アプリの離脱率を劇的に下げる。あと敵キャラのデザインはユーモアが命。「Nullぽガイコツ」が出てきたら思わずスクショしてSNSに貼りたくなる。それが無料の集客チャネルになる。

## 2. ターゲットと学習内容

**白鳥**: ターゲットは「プログラミング学習を始めたが挫折しかけている初学者」に絞る。具体的にはJavaScriptの基礎文法を学び始めた層。出題範囲は変数、データ型、条件分岐、ループ、関数、配列、オブジェクトの7分野。1問は4択で、読んで10秒、考えて20秒で答えられる粒度にする。長文コードを読ませる問題は初期には入れない。認知負荷が戦闘のテンポを殺すから。

**綾瀬**: その7分野、そのままエリア構成に使えるな。「はじまりの村(変数)」→「条件分岐の森」→「ループの洞窟」→「関数の塔」→「配列の海」→「オブジェクトの城」…と、ドラクエ的な世界地図に学習カリキュラムをマッピングする。エリアボスを倒すと次のエリアが解放される。**カリキュラム進行=物語進行**という一致が、このゲームの発明だ。

**桜庭**: セッション設計は「1プレイ3分」を目標に。通勤電車で1戦闘できるサイズ感。1戦闘=クイズ3〜5問。これなら「ちょっとだけ」が積み重なる。

## 3. コアループの定義

**綾瀬**: コアループを板書する。

1. フィールドを歩く(方向キー/タップ移動)
2. 敵とエンカウント(ランダム+シンボル)
3. コマンドバトル画面へ遷移
4. クイズ出題 → 正解なら攻撃ヒットで敵HP減、不正解なら敵の反撃で自HP減+解説表示
5. 敵撃破で経験値・ゴールド獲得 → レベルアップ演出
6. フィールドに戻る → ボス撃破で次エリア解放

**白鳥**: 4のところ、不正解時の解説表示は**次の行動を選べない強制読了1.5秒**くらいのウェイトを入れてほしい。読み飛ばし防止よ。それと同じ問題が後で再出題される**間隔反復**を入れたい。間違えた問題は出題プールの重みを上げる。

**九条**: 重み付き出題プールね、実装は軽い。問題データはJSONで持って、各問題に分野タグと難易度を付ける。敵ごとに「出題する分野と難易度帯」を定義すれば、エリアとカリキュラムの対応が自然に取れる。

## 4. スコープ議論(初回リリースで削るもの)

**九条**: 削る勇気の時間だ。俺の提案:
- **削る**: マルチプレイ、アカウント登録、実コード記述問題、BGM(著作権と容量)、モバイルアプリ化
- **残す**: フィールド移動、戦闘、クイズ、レベル・HP・経験値、エリア解放、localStorage保存、効果音(Web Audio APIで合成)

**桜庭**: BGM削るのは賛成だけど、効果音は絶対残して。攻撃ヒット音とレベルアップ音がないRPGは死んでるのと同じ。Web Audio APIの矩形波でファミコン風の音を合成すれば、音声ファイルゼロでいける。

**綾瀬**: 実コード記述は第2弾でいい。ただし4択の選択肢は「よくある間違い」をディストラクタ(誤答選択肢)として設計してくれ。白鳥、頼めるか。

**白鳥**: 任せて。誤概念(misconception)ベースのディストラクタ設計は私の専門よ。例えば `==` と `===` の混同、`let` と `const` の再代入、0始まりインデックスの取り違え。間違いのパターンにこそ教育価値が宿るの。

## 5. 差別化と成功指標

**桜庭**: 競合はProgateやドリル系アプリだけど、あちらは「教材にゲーム要素を足した」もの。うちは「ゲームが先、教材が中身」。**ゲームとして手触りが良いこと**が最大の差別化。KPIは (1) 初回セッションで戦闘3回以上完了率50%、(2) 翌日再訪率20%、(3) 1エリアクリア率40% を仮置きする。

**白鳥**: 学習効果の指標も。同一分野の正答率がプレイ時間とともに上がるかをlocalStorage内で追える設計にして。効果が示せれば教育機関にも売り込める。

**九条**: 全部フロントで計測できる範囲だな。よし、技術スタックは **Vite + TypeScript + Canvas API、フレームワークなし**。ゲームループは requestAnimationFrame、状態管理は自前の軽量ステートマシン。ビルド成果物は静的ファイルのみでVercelに即デプロイ。

## 6. 決定事項

1. プロダクト名は **「コードクエスト(CodeQuest)」**
2. ジャンル: 2Dピクセルアート・コマンドバトルRPG風 学習アプリ(Web)
3. ターゲット: JavaScript初学者(挫折しかけ層)
4. コアループ: 探索→エンカウント→クイズバトル→経験値→レベルアップ→エリア解放
5. 学習設計: 4択クイズ/即時解説/誤概念ベースの誤答選択肢/間違えた問題の重み付き再出題
6. 技術: Vite + TypeScript + Canvas、画像・音声アセット不使用(コード生成)、localStorage保存、Vercel静的デプロイ
7. 初回スコープ: 3エリア(変数の村・条件分岐の森・ループの洞窟)+ボス戦、問題数30問以上
8. 次アクション: 設計打ち合わせを本日中に実施(全員参加)

**綾瀬**: いい会議だった。ドラクエを遊んで育った俺たちが、次の世代に「学ぶことは冒険だ」と伝える番だ。解散!
