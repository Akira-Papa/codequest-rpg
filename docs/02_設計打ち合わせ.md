## 1. 技術スタック(確定)

| 項目 | 選定 | 理由 |
|---|---|---|
| ビルド | Vite 6 + TypeScript | 高速・設定最小・静的出力 |
| 描画 | Canvas 2D API | ピクセルアートをコードで生成、画像アセット不要 |
| 音 | Web Audio API(矩形波・三角波合成) | ファミコン風SE、音声ファイル不要 |
| 状態管理 | 自前ステートマシン | 依存ゼロ、シーン遷移が明確 |
| 永続化 | localStorage | 認証不要、即保存 |
| デプロイ | Vercel(静的サイト) | `vite build` の dist/ をそのまま配信 |
| フレームワーク | なし | 依存最小化・バンドル軽量化 |

## 2. アーキテクチャ

```
src/
├── main.ts            # エントリ。Canvas初期化、ゲームループ起動
├── game.ts            # Game クラス:シーン管理・ループ(rAF)
├── scenes/
│   ├── title.ts       # タイトル画面(はじめから/つづきから)
│   ├── field.ts       # フィールド探索(タイルマップ・移動・エンカウント)
│   └── battle.ts      # クイズバトル(コマンド→出題→判定→演出→レベルアップ/エリア解放)
├── core/
│   ├── input.ts       # キーボード+タッチ入力の抽象化
│   ├── audio.ts       # 効果音シンセ(attack/correct/wrong/levelup/fanfare)
│   ├── save.ts        # localStorage セーブ/ロード(バージョン付きスキーマ)
│   └── logic.ts       # 純関数(経験値計算・出題選択・重み付け) ※テスト対象
├── data/
│   ├── questions.ts   # クイズデータ(分野タグ・難易度・解説付き)
│   ├── enemies.ts     # 敵定義(名前・ドット絵・HP・出題分野・難易度帯)
│   └── areas.ts       # エリア定義(マップ・敵テーブル・ボス・解放条件)
├── render/
│   ├── sprites.ts     # ドット絵データ(2次元配列→Canvas描画)
│   └── ui.ts          # ウィンドウ枠・HPバー・テキスト描画(DQ風UI)
└── types.ts           # 共有型定義
```

**九条**: シーンは `Scene` インターフェース(`update(dt)` / `draw(ctx)` / `onEnter` / `onExit`)で統一。`Game` がスタックではなく単一カレントシーンを持つ。シンプルさ優先。

## 3. 主要データ設計

### 3.1 クイズ問題

```ts
interface Question {
  id: string;
  category: 'variable' | 'condition' | 'loop';  // 初回3分野
  difficulty: 1 | 2 | 3;
  text: string;          // 問題文(コード片含む)
  choices: string[];     // 4択。誤概念ベースのディストラクタ
  answer: number;        // 正解インデックス
  explanation: string;   // 1〜2行の解説(正誤どちらでも表示)
}
```

**白鳥**: 30問以上。分野×難易度でマトリクスを埋める。間違えた問題は `wrongCount` をセーブデータに記録し、出題重み = `1 + wrongCount * 2` で再出題されやすくする(間隔反復の簡易実装)。

### 3.2 敵・エリア

```ts
interface Enemy {
  id: string; name: string;
  sprite: number[][];        // パレットインデックスの2次元配列
  palette: string[];
  maxHp: number;             // 正解n回で倒せるHP設計
  attack: number;            // 不正解時にプレイヤーが受けるダメージ
  categories: Category[];    // 出題分野
  difficulty: [number, number]; // 出題難易度帯
  exp: number; isBoss?: boolean;
}
```

エリアは3つ:**変数の村(周辺草原)/条件分岐の森/ループの洞窟**。各エリアに雑魚2種+ボス1種。ボス撃破で次エリア解放。

### 3.3 プレイヤー/セーブデータ

```ts
interface SaveData {
  version: 1;
  player: { level: number; exp: number; maxHp: number; hp: number; x: number; y: number };
  unlockedAreas: string[];
  currentArea: string;
  defeatedBosses: string[];
  stats: Record<string, { asked: number; correct: number; wrongCount: number }>; // questionId毎
}
```

- レベルアップ必要経験値: `next = 20 * level^1.5`(切り上げ)
- レベルアップ時: maxHp +5、全回復
- HP0時: ゲームオーバーではなく「村に戻される」(HP全回復、経験値保持)——白鳥の「失敗の安全性」原則

## 4. バトルフロー(状態遷移)

```
INTRO(敵出現メッセージ)
 → COMMAND(たたかう/にげる)
 → QUESTION(出題・4択・キー1-4/クリック)
 → JUDGE(正解: 敵ダメージ演出+SE / 不正解: 自ダメージ+解説を1.5秒強制表示)
 → (敵HP0 → VICTORY: 経験値獲得→必要ならLEVELUP演出)
 → (自HP0 → DEFEAT: 村へ帰還)
 → COMMANDへ戻る
```

**綾瀬**: 演出優先度は (1)ヒット時の敵フラッシュ+画面シェイク、(2)レベルアップのファンファーレ+ステータス上昇表示、(3)ボス撃破時のエリア解放メッセージ。この3つは削らない。

**桜庭**: UIはドラクエ準拠——黒背景+白枠角丸ウィンドウ+白文字。フォントは `"Press Start 2P"` 系は重いのでシステムのmonospaceで代用し、`image-rendering: pixelated` でCanvasを2〜3倍拡大表示。カラーパレットはファミコン準拠の落ち着いた色(背景 #0f0f23、UI枠 #ffffff、強調 #f7d51d)。

## 5. フィールド設計

- タイルマップ: 16×12タイル、1タイル16px(論理)→表示3倍拡大
- タイル種: 草/木(通行不可)/水(通行不可)/道/村・洞窟入口(エリア移動)
- 移動: グリッド単位、矢印キー/WASD + 画面タップ(モバイル: 画面4分割方向判定)
- エンカウント: 草タイル歩行時 12%/歩。ボスはシンボル配置(接触で戦闘)
- **九条**: 当たり判定はタイル配列参照のみ。物理エンジン不要。

## 6. 効果音設計(Web Audio)

| SE | 波形 | 内容 |
|---|---|---|
| カーソル移動 | 矩形波 | 短いピッ |
| 攻撃ヒット | ノイズ+矩形波下降 | ザシュッ |
| 正解 | 矩形波上昇2音 | ピロリン |
| 不正解 | 三角波下降 | ブブー |
| レベルアップ | 矩形波アルペジオ | ファンファーレ風4音 |

初回ユーザー操作で `AudioContext` を resume(自動再生制限対応)。

## 7. 非機能要件

- **パフォーマンス**: バンドル < 100KB(gzip)、60fps維持、初回表示 < 1s
- **対応環境**: モダンブラウザ(Chrome/Safari/Firefox/Edge 最新2版)、スマホ縦横対応
- **アクセシビリティ**: クイズはキーボードのみで完結操作可、色だけに依存しない正誤表示(○×記号併用)
- **セーブ**: 戦闘終了・エリア移動・レベルアップ時に自動保存。スキーマに `version` を持たせ将来のマイグレーションに備える

## 8. テスト方針

**白鳥**: 教育ロジック(出題重み・レベル計算)は純関数に切り出してユニットテスト可能に。
**九条**: 初回はVitestで `save.ts` / 出題選択 / 経験値計算をテスト。描画はレビューと手動確認で担保。

## 9. マイルストーン

1. **M1**: プロジェクト雛形+ゲームループ+タイトル画面

## 10. 実装時の設計変更(レビュー反映済み)

実装とレビュー(鬼教官4名)を経て、以下を当初設計から変更した。実装が正。

| 項目 | 当初設計 | 最終実装 | 理由 |
|---|---|---|---|
| フォント | システムmonospace | Google Fonts「DotGothic16」 | 日本語ピクセルフォントの体験価値が大きく、読み込み失敗時もmonospaceにフォールバック(最大1.5秒待ち)するため採用 |
| エンカウント率 | 12%/歩 | 草原・森14%、洞窟16% | プレイテストでのテンポ調整 |
| ドット絵の型 | `number[][]`+`string[]` | `string[]`+`Record<string,string>` | 文字マップの方が目視編集しやすい |
| ゴールド | コアループに記載 | **v1では未実装**(使い道の店が無いため) | YAGNI。ショップ実装時に導入 |
| レベルアップ演出 | scenes/levelup.ts | battle.ts のメッセージフェーズに統合 | シーン分割が過剰だった |
| セーブデータ | 記載スキーマ | `cleared`(クリアフラグ)を追加 | エンディング後の状態表現 |
| ボス撃破時 | 記載なし | **HP全回復**を追加 | エリア2・3に回復手段が無く「わざと死ぬのが最適解」になる設計崩壊のレビュー指摘(桐生)対応 |
| ボス戦導入 | 記載なし | 推奨レベル未満なら警告メッセージ | 低レベル特攻の導線不足指摘(桐生)対応 |
| タップ操作 | 画面4分割方向判定 | プレイヤー相対方向+長押し連続移動 | モバイル操作性指摘(桐生)対応 |
| メニュー/選択肢タップ | 1タップ実行 | 2段階タップ(1回目選択・2回目確定)+「はじめから」に上書き確認 | 誤タップでセーブ消滅する事故防止(轟・桐生)対応 |
| 間隔反復 | wrongCount重み付け | 同左+**正解時に重みを減衰** | 結線漏れバグ(轟指摘)の修正と合わせて改善 |
| 進行ロジック | シーン内実装 | `core/logic.ts` に純関数化(`resolveVictory`/`resolveDefeat`/`recordAnswer`)+テスト | テスト可能性指摘(九鬼)対応 |
